## 실수-M1: "더 잘 해줘" 루프

**상황:**
```
사용자: 코드 짜줘
Claude: [코드 작성]
사용자: 더 좋게 만들어줘
Claude: [코드 수정]
사용자: 더 최적화해줘
Claude: [또 수정]
```

**문제:** 기준이 없으면 Claude도 방향을 못 잡음 → 계속 다른 방향으로 수정 → 토큰 낭비 + 결과 악화 가능

**해결:** 처음부터 기준을 명시하라.
```
"이 코드를 최적화해줘. 기준: 가독성 > 성능. 변수명은 영어로."
```

## 실수-M2: 대화 중간에 주제 전환

**상황:**
```
[긴 파이썬 디버깅 대화 중...]
사용자: "아 그리고 CSS 질문 하나만요..."
```

**문제:** 파이썬 디버깅 히스토리 전체가 CSS 질문의 input에 포함됨
→ 전혀 관계없는 토큰 수천 개 낭비

**해결:** `/clear` 후 새 질문. 맥락이 필요하면 한 줄로 요약해서 새 대화에서 시작.

## 실수-M3: Claude를 파일 시스템처럼 쓰기

**상황:**
```
사용자: "아까 짜준 코드 다시 보여줘"
사용자: "저번에 만든 함수 기억해?"
```

**문제:** Claude는 대화 히스토리 안에서만 기억함.
새 대화를 시작하면 이전 내용은 완전히 없어짐.

**해결:** 재사용할 코드는 파일로 저장하거나 히스토리가 있는 같은 대화에서 요청.
중요한 결과물은 항상 직접 파일에 저장.

## 실수-M4: 긴 출력을 무조건 요청하기

**상황:**
```
사용자: "이 주제에 대해 자세히 설명해줘"
사용자: "완전한 코드로 작성해줘"
사용자: "모든 케이스를 다 다뤄줘"
```

**문제:** "자세히", "완전한", "모든" → Claude가 최대한 길게 씀 → output 토큰 폭증
대부분은 그 내용을 다 읽지도 않음.

**해결:** 필요한 수준을 명시하라.
```
"핵심만 3줄로"
"happy path만, 에러 처리는 제외"
"가장 중요한 케이스 2개만"
```

## 실수-M5: 에러 메시지 없이 "안 된다"고만 하기

**상황:**
```
사용자: "코드 실행했는데 안 돼요"
사용자: "에러 났어요. 왜 그럴까요?"
```

**문제:** Claude가 추측으로 여러 가능성을 나열 → 긴 output, 맞지 않을 가능성 높음

**해결:** 에러 메시지를 그대로 붙여넣기.
```
"아래 에러 발생. 원인과 수정법 한 줄씩:
TypeError: unsupported operand type(s) for +: 'int' and 'str'
  File "main.py", line 23, in calculate"
```

## 실수-M6: 승인 없이 계속 진행시키기

**상황:**
```
사용자: "로그인 기능 전체 만들어줘"
→ Claude가 DB 스키마, API, 프론트엔드, 테스트까지 알아서 다 작성
→ 알고 보니 이미 DB 스키마는 있었음
```

**문제:** 범위를 안 정하면 Claude가 임의로 범위를 확장 → 불필요한 코드 + 기존 것과 충돌

**해결:** 단계를 나눠서 각 단계마다 확인하며 진행.
```
"로그인 기능 구현 계획만 먼저 짜줘. 코드 작성은 내가 승인 후에."
```

## 실수-M7: 모든 작업을 Claude에게 맡기기

**상황:**
- 간단히 검색하면 되는 것을 Claude에게 질문
- 공식 문서에 있는 것을 Claude에게 설명 요청
- 직접 실행해보면 되는 것을 Claude에게 결과 예측 요청

**문제:** Claude는 최신 정보를 모를 수 있고, 정확한 수치나 버전 정보는 환각 가능성 있음

**해결:**
```
검색으로 해결될 것 → 검색
공식 문서가 있는 것 → 공식 문서
실행해볼 수 있는 것 → 직접 실행
Claude는 "조합과 판단"이 필요한 작업에 써라
```
