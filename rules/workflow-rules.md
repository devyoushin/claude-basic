# 작업 유형별 워크플로우 규칙

> 카테고리: 워크플로우
> 관련 문서: ../claude-guide.md, agent-rules.md

---

## 개념: 왜 워크플로우가 필요한가?

같은 Claude도 **어떤 순서로 요청하느냐**에 따라 결과 품질과 토큰 소모가 크게 달라진다.
작업 유형별로 검증된 순서를 따르면 재시도가 줄어든다.

---

## RULE-W1: 버그 수정 워크플로우

```
1. [현상 설명] 어떤 에러가 어떤 상황에서 발생하는지
2. [범위 지정] 관련 파일의 해당 줄만 읽혀주기
3. [원인 분석] "이 코드에서 에러 원인만 찾아줘" (수정 요청 X)
4. [수정 확인] 원인 파악 후 "이 방향으로 수정해줘"
```

**하지 말 것:** "이 파일 읽고 버그 찾아서 고쳐줘" — 한 번에 다 시키면
Claude가 전체를 읽고, 추측으로 수정하고, 설명까지 붙임 → 토큰 낭비 + 엉뚱한 수정

---

## RULE-W2: 새 기능 개발 워크플로우

```
1. [설계 먼저] "아래 요구사항으로 구현 계획을 단계별로 작성해줘. 코드 작성 X"
2. [계획 검토] 계획이 맞으면 승인, 틀리면 이 단계에서 수정 (코드 수정보다 훨씬 저렴)
3. [단계별 구현] 승인된 계획을 단계별로 하나씩 구현 요청
4. [검증] 각 단계 완료 후 확인하고 다음 단계로
```

**핵심:** 설계 단계에서 방향을 잡아야 코드 재작성을 피할 수 있다.
코드 100줄 재작성 >> 계획 텍스트 10줄 수정 (토큰 차이 10배 이상)

---

## RULE-W3: 코드 리뷰 워크플로우

```
1. [범위 지정] 리뷰받을 파일/함수 범위를 명확히
2. [관점 지정] "보안", "성능", "가독성" 중 무엇을 볼지 지정
3. [형식 지정] "심각도 순으로 목록만, 설명 없이"
```

**예시:**
```
auth.py의 login() 함수(40~80줄)를 보안 관점으로 리뷰해줘.
형식: [심각도: HIGH/MED/LOW] 문제 한 줄 요약
설명과 수정 코드는 없이.
```

**이유:** 관점 없는 리뷰 요청은 Claude가 모든 것을 다 지적하고 수정 코드까지 작성 → output 폭증

---

## RULE-W4: 문서/코드 이해 워크플로우

낯선 코드베이스를 파악할 때:

```
1. [구조 파악] Explore Agent에게 위임: "이 프로젝트 구조와 주요 파일 역할 파악"
2. [진입점 파악] "가장 중요한 파일 3개만 알려줘"
3. [핵심 로직] 그 파일들만 읽기
4. [질문] 파악한 맥락 위에서 구체적 질문
```

**하지 말 것:** 처음부터 모든 파일을 읽히는 것.
전체 파악보다 핵심 흐름 파악이 먼저.

---

## RULE-W5: 반복 작업 자동화 워크플로우

같은 패턴의 작업을 여러 번 해야 할 때:

```
1. [예시 1개] 첫 번째 항목으로 원하는 결과물 작성
2. [패턴 확인] 결과물이 맞는지 확인
3. [일괄 처리] "이 패턴으로 나머지 N개도 처리해줘" + 데이터만 전달
```

**예시:**
```
[1단계] "이 JSON을 SQL INSERT문으로 변환해줘: {item1}"
[2단계] 결과 확인 후 OK
[3단계] "같은 방식으로 아래 10개 모두 변환해줘: [item2~item11]"
```

**이유:** 패턴을 먼저 확정하면 재시도 없이 일괄 처리 가능.
10개를 하나씩 요청하면 10번의 왕복 = 10배 토큰.
