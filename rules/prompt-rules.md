# 프롬프트 작성 규칙

> 카테고리: 프롬프트 엔지니어링
> 참고 예시: ../token-comparison.md

---

## RULE-P1: 역할을 부여하면 품질이 올라간다

**규칙:** 복잡한 작업엔 Claude에게 역할을 먼저 지정하라.

```
"당신은 10년 경력 시니어 백엔드 개발자입니다."
"당신은 한국어-영어 전문 번역가입니다."
"당신은 코드 보안 감사 전문가입니다."
```

**주의:** 단순 질문엔 역할 부여가 오히려 토큰 낭비

---

## RULE-P2: 예시를 보여주면 원하는 형식이 나온다 (Few-shot)

**규칙:** 원하는 출력 패턴이 있으면 예시 1~2개를 먼저 보여줘라.

```
예시 입력: "2026-04-16"
예시 출력: "April 16, 2026 (Wednesday)"

이 형식으로 변환: "2026-12-25"
```

---

## RULE-P3: "단계별로 생각해줘"는 복잡한 문제에만 써라

**규칙:** 추론이 필요한 문제에만 CoT(Chain of Thought)를 유도하라.

```
✅ 적합: 알고리즘 설계, 버그 원인 분석, 아키텍처 결정
❌ 낭비: 단순 번역, 포맷 변환, 짧은 코드 작성
```

---

## RULE-P4: 부정문보다 긍정문으로 지시하라

**규칙:** "~하지 마라" 대신 "~해라"로 써라.

```
❌ "너무 길게 설명하지 마세요"
✅ "3줄로 요약해주세요"

❌ "전문 용어 쓰지 마세요"
✅ "중학생도 이해할 수 있게 설명해주세요"
```

---

## RULE-P5: CLAUDE.md로 반복 설명을 없애라

**규칙:** 매 대화마다 반복하는 컨텍스트는 프로젝트 루트의 `CLAUDE.md`에 저장하라.

**CLAUDE.md 예시:**
```markdown
# 프로젝트 컨텍스트
- 언어: Python 3.11
- 프레임워크: FastAPI
- DB: PostgreSQL
- 코드 스타일: Black formatter, 타입 힌트 필수
- 테스트: pytest, 커버리지 80% 이상
```

**이유:** Claude Code가 자동으로 읽어서 매 대화에 주입 → 설명 반복 없음
