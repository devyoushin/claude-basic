# 토큰 절약 규칙

> 카테고리: 토큰 관리
> 참고 예시: ../token-comparison.md

---

## RULE-T1: 출력 형식을 항상 명시하라

**규칙:** 질문할 때 원하는 출력 형식을 함께 써라.

**적용:**
- `표로 정리해줘`
- `3줄 이내로`
- `코드만, 설명 없이`
- `JSON으로`
- `한 줄 요약`

**이유:** 형식 미지정 시 Claude는 장문으로 응답 → output 토큰 최대 85% 낭비

---

## RULE-T2: 새 주제는 새 대화에서 시작하라

**규칙:** 이전 질문과 관련 없는 새 주제는 반드시 `/clear` 후 질문하라.

**이유:** 대화 히스토리 전체가 매 턴마다 input에 포함됨
→ 5번째 질문이면 1~4번 대화가 통째로 다시 전송됨

**Claude Code 명령:**
```
/clear        # 히스토리 초기화
/compact      # 히스토리를 요약으로 압축 (맥락 유지하고 싶을 때)
```

---

## RULE-T3: 파일은 필요한 범위만 읽혀라

**규칙:** 파일 전체를 읽히지 말고 줄 번호나 함수명으로 범위를 지정하라.

```
❌ "main.py 읽고 버그 찾아줘"
✅ "main.py의 50~100줄만 읽고 버그 찾아줘"
✅ "main.py에서 calculate_tax 함수만 확인해줘"
```

---

## RULE-T4: 조건을 한 번에 완전하게 써라

**규칙:** 모호하게 요청하지 말고, 조건/제약/형식을 첫 메시지에 다 담아라.

**템플릿:**
```
[목적]: 무엇을 만들거나 분석할지
[조건]: 언어, 프레임워크, 제약사항
[입출력]: 입력값과 출력값 형태
[형식]: 코드만 / 설명 포함 / 표 등
```

**이유:** 모호한 요청은 되묻기를 유발 → 왕복 1회당 토큰 2배

---

## RULE-T5: 모델을 작업 난이도에 맞게 선택하라

**규칙:** 단순 작업에 고성능 모델을 쓰지 마라.

| 작업 유형 | 추천 모델 |
|-----------|-----------|
| 번역, 요약, 단순 질문 | Haiku 4.5 |
| 코딩, 분석, 일반 작업 | Sonnet 4.6 (기본) |
| 복잡한 추론, 아키텍처 설계 | Opus 4.6 |
